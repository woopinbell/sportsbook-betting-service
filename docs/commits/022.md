## 작업 내역

### WireMock으로 betting orchestration만 격리

risk/wallet을 항상 approve/debit하는 WireMock으로 대체해, 벤치마크가 **betting 자신의 orchestration 비용**만 측정하게 한다(의존 서비스 성능에 오염되지 않게). 한 요청 = 전체 ADR-0017 접수: validate → slippage(Redis) → risk(HTTP) → wallet(HTTP) → accept → outbox insert. `seed.sh`가 고정 selection의 odds 캐시를 미리 채운다.

### 왜 dev-host에서 p99 < 100ms는 도달 불가인가 (정직한 분석)

placement 1건은 **동기 HTTP 2회 + DB 트랜잭션 2개 + Redis + Avro 인코드 + outbox insert**다. dev 호스트는 PostgreSQL·Kafka·WireMock·betting JVM이 한 Docker VM의 CPU를 나눠 써서 **median이 이미 ~90ms**다. 150 RPS를 넘기면 constant-arrival 생성기가 단일 VM을 추월 — VU가 쌓이고 내부 호출이 500ms read timeout을 치며 saturation 아티팩트가 된다(latency 측정이 아님).

핵심 판단: **하드웨어 한계지 구조적 천장이 아니다.** 아키텍처(외부 호출을 DB 트랜잭션 밖에, breaker fail-fast, 멱등 재시도)에 병목이 없다. p99 < 100ms / 1만 동시는 multi-replica·전용 호스트의 production 목표(orchestration e2e에서 실측). dev 박스에서는 **정합성 불변식**을 증명하는 데 집중했다 — 단일 수락/단일 차감은 `BetPlacementIntegrationTest`(015)가 권위 있게 증명.

### tuning: httpclient5 풀링 시도 → revert (박제)

risk/wallet 연결을 풀링하려 `httpclient5`를 시도했는데, **default 5-per-route 풀**이 동시 placement 경로를 starve시켜 p99가 초 단위로 튀었다 → revert. default JDK `HttpClient`(Spring Boot 기본)가 이미 풀링하고 더 나은 숫자를 줬다. 교훈: **잘못 *크기 잡은* Apache 풀은 default보다 나쁘다** — 제대로 sizing한 풀이 production 옵션이지 default가 아니다. median이 connection-bound가 아니라 CPU-bound라 풀링이 floor를 못 움직였다.
