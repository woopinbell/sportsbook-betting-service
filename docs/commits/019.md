# 019 — reconciliation 테스트 (roll-forward/back/defer) (`test(reconciliation): roll-forward / roll-back / defer`)

## 개요 — 왜 이 변경이 필요한가

018의 세 분기는 wallet 응답에 따라 갈린다. stale PENDING bet을 만들고 wallet stub의 응답을 바꿔가며 각 분기를 실 인프라로 박는다.

## 작업 순서

1. `@SpringBootTest`(PostgreSQL/Redis/WireMock wallet, Kafka 제외) + stale PENDING(past createdAt, pending-timeout 0s).
2. wallet 200 / 422 / 503 케이스.

## 작업 내역

`pending-timeout`을 0s로, bet의 `createdAt`을 과거로 만들어 즉시 stale로 보이게 한 뒤 잡을 구동:

- **wallet 200 → ACCEPTED + outbox 행 1개**(roll forward),
- **wallet 422 insufficient → REJECTED, reason `RECONCILED_NO_DEBIT`, outbox 없음**(roll back),
- **wallet 503 → PENDING 유지**(defer, 다음 tick).

roll-back 케이스에서 outbox 행이 없음을 확인하는 게 중요 — 거절은 이벤트를 내보내지 않는다. roll-forward에서 정확히 1개 outbox 행은 018의 PENDING 가드가 중복을 막음을 방증.

## 결과

018의 세 분기가 실 인프라(WireMock wallet 응답별)로 증명됨.

## 요약

- stale PENDING(timeout 0s, past createdAt)으로 roll-forward(200)/roll-back(422)/defer(503) 검증.
- roll-back은 REJECTED + `RECONCILED_NO_DEBIT`, outbox 없음. roll-forward는 outbox 1개.
