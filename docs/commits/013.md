# 013 — outbox write side + Avro (`feat(outbox): transactional outbox write side for BetPlacedRequested`)

## 개요 — 왜 이 변경이 필요한가

012의 테이블에 쓰는 쪽. `OutboxEvent` 엔티티 + `AvroSerializer`(Schema Registry 없는 binary) + `BetEventFactory`(수락된 Bet → `BetPlacedRequested` Avro). publisher(016)는 다음 작업이고, 여기는 **write side만**.

## 작업 순서

1. `OutboxEvent` 엔티티 + repository.
2. `AvroSerializer` — SpecificRecord binary encode/decode.
3. `BetEventFactory` — Bet → `BetPlacedRequested` + partition key 결정.

## 작업 내역

### Avro binary, Schema Registry 없이 (ADR-0014)

```java
public static <T extends SpecificRecord> byte[] serialize(T record) {
  DatumWriter<SpecificRecord> writer = new SpecificDatumWriter<>(record.getSchema());
  BinaryEncoder encoder = EncoderFactory.get().binaryEncoder(out, null);
  writer.write(record, encoder);
  encoder.flush();
  return out.toByteArray();
}
```

V1은 Schema Registry 없이 발행한다(ADR-0014). 각 소비자가 **같은 shared-protocol 생성 클래스를 pin**하므로 schema id가 암묵적이다. encode boilerplate를 서비스 레이어에 새지 않게 이 헬퍼로 격리(decode는 현재 테스트의 payload 검증용).

> **C/C++ 관점**: Avro는 protobuf처럼 스키마 기반 binary 직렬화다. `SpecificRecord`는 `.avsc`에서 생성된 강타입 클래스(shared-protocol이 Maven 빌드 시 생성). Schema Registry는 보통 wire에 schema id를 박아 진화를 관리하지만, V1은 생략하고 양쪽이 같은 클래스 버전을 쓰는 것으로 대체.

### partition key = userId, eventId 아님

```java
return OutboxEvent.pending(UuidV7.generate(), TOPIC, bet.userId().toString(), SCHEMA, payload, now);
//                                                    ^^^^^^^^^^^^^^^^^^^^^^^ partition key
```

ADR-0006의 일반 규칙은 "partition key = eventId(한 경기의 모든 변경이 같은 partition)"인데, 여기선 **userId**를 쓴다. 이유: (1) 슬립은 *여러 경기*에 걸칠 수 있어 단일 eventId가 없다, (2) 소비자(risk의 per-user sliding window, settlement read model)가 **per-user 순서/co-location**을 원한다. ADR-0006의 eventId 규칙은 *per-match 이벤트 스트림*(odds, 정산)을 겨냥한 것이지, *per-user 접수 액션*이 아니다 — 그래서 의도적으로 갈라쓴다. 이 판단을 커밋 본문에 박아 둔 게 retrospective seed가 됐다.

### topic은 `.v1`

`bet.placed.v1` — `bet.*` 도메인 이벤트 family가 `.v1` 컨벤션을 쓴다. (Phase 5 통합에서 settlement/gateway가 이 family에 맞춰 `.v1`로 통일됐다 — 그 불일치는 betting 밖의 일.)

## 결과

수락된 Bet을 Avro `BetPlacedRequested` outbox 행으로 만드는 write side. 014가 접수 흐름에서 이걸 enqueue, 016이 drain.

## 요약

- Avro binary(Schema Registry 없음, ADR-0014), 소비자가 클래스 pin.
- **partition key = userId**(eventId 아님) — 슬립이 다중 경기 + 소비자가 per-user 순서 원함.
- topic `bet.placed.v1`.

## 기억/설명 Level

- **L3 (외워서 설명)** — *partition key가 왜 userId인가*: 슬립은 여러 경기에 걸쳐 단일 eventId가 없고, 소비자(risk 슬라이딩 윈도우/settlement read model)가 per-user 순서·co-location을 원한다. ADR-0006의 eventId 규칙은 per-match 스트림용이라 per-user 접수엔 userId로 갈라쓴다.
- **L2 (문서 보며 설명)** — Avro SpecificRecord binary + Schema Registry 생략(클래스 pin), `bet.placed.v1` family 네이밍.
- **L1 (읽으면 됨)** — OutboxEvent의 단일 mutable 필드는 published_at, 나머지 write-once.
