# 026 — 정산 consume·멱등·전이 가드 테스트 (`test(settlement): cover kafka consume, idempotency, and transition guards`)

## 개요 — 왜 이 변경이 필요한가

024/025의 정산 consumer는 **재배달 안전**(멱등)과 **불법 입력 차단**(상태전이 가드 → DLT)이 생명이다. 실 Avro 이벤트를 listener로 흘려 전이를 검증하고, 멱등/unknown 가드는 트랜잭션 서비스로 직접 구동해 결정적으로 박는다.

## 작업 순서

1. `SettlementConsumerIntegrationTest`(@EmbeddedKafka + Testcontainers) — 실 listener 경로.
2. 멱등/unknown 가드는 `BetSettlementService` 직접 호출(결정적).
3. `BetTest`에 SETTLED/VOIDED happy + 불법 전이 가드 추가.

## 작업 내역

### listener 경로는 EmbeddedKafka로, 가드는 서비스 직접

```
listener 경로(@EmbeddedKafka):  실 BetSettled/BetVoided Avro → listener → SETTLED(result+payout) / VOIDED(reason)
가드(서비스 직접 호출):          재적용 no-op(@Version 불변), unknown betId throw(DLT 전제)
```

정산 consumer 테스트는 016/017의 outbox 테스트와 같은 인프라 골격(@EmbeddedKafka + Testcontainers PostgreSQL/Redis)을 쓴다. 실 Avro record를 listener에 흘려 SETTLED(기록된 result+payout)/VOIDED(기록된 reason) 도달을 단언한다.

**멱등/unknown 가드는 트랜잭션 서비스를 직접 호출**해 검증한다 — Kafka 재배달 타이밍에 의존하면 비결정적이라, 서비스 메서드를 두 번 부르는 것으로 결정적으로 만든다:
- 두 번째 apply는 **no-op**(`@Version`이 안 바뀜 = DB write 없음),
- unknown betId는 **throw**(DLT precondition).

> **C/C++ 관점**: `@Version`(JPA optimistic lock 컬럼)이 안 바뀌었다는 건 그 트랜잭션이 **DB를 안 건드렸다**는 직접 증거다 — 멱등 no-op을 "버전 불변"으로 검증하는 게 깔끔하다. 두 번째 호출이 진짜 아무 일도 안 했음을 부수효과(버전)로 확인.

### 도메인 전이 가드 (BetTest)

`BetTest`에 ACCEPTED→SETTLED/VOIDED happy path(zero-payout LOST 포함)와 **불법 전이 가드**를 추가: accept 전 settle/void, double settle, settle 후 void, payout currency 불일치 → 전부 throw. 024/025의 전이가 객체 수준에서도 안전함을 박는다.

## 결과

024/025의 consume·멱등·전이 가드가 실 Kafka 경로 + 결정적 서비스 호출 + 객체 수준 전이로 삼중 검증됨. Phase 4 정산 consumer 완성.

## 요약

- listener 경로는 @EmbeddedKafka로 SETTLED/VOIDED 도달 검증.
- 멱등/unknown 가드는 서비스 직접 호출(결정적): 재적용 no-op(@Version 불변), unknown throw(DLT 전제).
- `BetTest`에 SETTLED/VOIDED happy(zero-payout LOST 포함) + 불법 전이 가드(double settle, void after settle, currency 불일치).
