# 024 — BetSettled consume → SETTLED (`feat(settlement): consume BetSettled, flip an accepted bet to SETTLED`)

## 개요 — 왜 이 변경이 필요한가

Phase 4의 핵심. ADR-0017이 "정산(SETTLED 전환)은 Phase 4에서 `BetSettled` consume으로"라 명시했다. settlement-service가 MatchResult 후 `BetSettled`(Avro)를 발행하면, betting이 그걸 consume해 ACCEPTED→SETTLED로 전이하고 실제 payout을 기록한다(ADR-0006 비동기 정산).

## 작업 순서

1. `V3__settlement_outcome.sql` — settlement_result + settled_payout_* + resolved_at.
2. `Bet.settle(result, payout, now)` 전이.
3. `BetSettlementService` — 멱등 + 상태전이 가드.
4. `SettlementResultListener` — Kafka consume(Avro 디코드).

## 작업 내역

### V3 컬럼: maxPayout과 settledPayout은 다르다

```sql
ALTER TABLE bet
    ADD COLUMN settlement_result       VARCHAR(8),   -- WON/LOST/PUSH/VOID (ADR-0013)
    ADD COLUMN settled_payout_amount   BIGINT,
    ADD COLUMN settled_payout_currency VARCHAR(3),
    ADD COLUMN resolved_at             TIMESTAMP WITH TIME ZONE;
```

수락 시 저장한 `max_payout`(V1)은 *최악 시* 감사용 수치고, `settled_payout`은 settlement이 결과에 따라 산정한 **실제 지급액**이다 — 둘을 분리해 기록한다(LOST면 settled_payout=0). check 제약으로 결과 유효성·payout 쌍(amount/currency 동시 null 또는 동시 set)·비음수·currency=stake_currency(슬립 내 FX 없음)를 박는다. `resolved_at`은 terminal 전이 시각(VOIDED와 공유, V4).

### 멱등 by betId + 상태전이 가드

```java
@Transactional
public void applySettled(UUID betId, SettlementResult result, Money payout, Instant settledAt) {
  Bet bet = bets.findById(betId)
      .orElseThrow(() -> new IllegalStateException("BetSettled for unknown betId " + betId));
  if (bet.status() == BetStatus.SETTLED) {     // 재배달 → no-op
    return;
  }
  bet.settle(result, payout, settledAt);       // ACCEPTED → SETTLED (아니면 throw)
}
```

Kafka는 **at-least-once**라 같은 `BetSettled`가 재배달된다. 가드의 경계가 핵심:
- 이미 SETTLED인 bet → **no-op**(멱등 — 재배달 안전),
- **unknown betId**나 **불법 전이**(ACCEPTED 아닌 상태) → **throw**.

throw하면 record가 retry → `<topic>.DLT`로 가서 **조용히 상태를 망가뜨리는 대신** 운영이 본다. "재배달이면 무시, 진짜 이상이면 DLT"의 명확한 경계.

### Avro를 aggregate에 안 새게 (edge에서만)

```java
@KafkaListener(topics = SETTLED_TOPIC, autoStartup = AUTO_STARTUP)   // bet.settled.v1
public void onBetSettled(byte[] payload) {
  BetSettled event = AvroSerializer.deserialize(payload, BetSettled.class);
  service.applySettled(UUID.fromString(...), SettlementResult.valueOf(...), toMoney(...), event.getSettledAt());
}
```

리스너(edge)에서만 Avro 타입을 다루고, 서비스엔 **도메인 값**만 넘긴다 — wire schema가 JPA aggregate로 새지 않게. producer의 byte[] 와이어 모양을 그대로 미러(Schema Registry 없음, ADR-0014). `autoStartup`이 프로퍼티라 브로커 없는 통합 테스트는 리스너를 주차(016의 주차 잡과 동형). topic은 `bet.settled.v1`.

> **C/C++ 관점**: Kafka DLT(Dead Letter Topic)는 처리 실패한 메시지를 버리지 않고 별도 토픽에 모으는 곳이다 — 큐의 "처리 불가 메시지" 격리. 멱등 가드 덕에 retry가 안전하고(같은 결과 재적용은 no-op), 진짜 이상(unknown/불법 전이)만 DLT로 가 사람이 본다.

## 결과

정산된 베팅이 SETTLED로 전이하고 실제 payout이 기록됨. at-least-once 재배달 안전, 이상은 DLT로.

## 요약

- V3로 settled_payout(실제 지급, maxPayout과 별개) + settlement_result + resolved_at 추가, check 제약.
- **멱등 by betId**(이미 SETTLED면 no-op) + **상태전이 가드**(unknown/불법 전이 → throw → DLT).
- Avro는 listener edge에서만, 서비스엔 도메인 값(wire schema 비누설). topic `bet.settled.v1`.

## 기억/설명 Level

- **L3 (외워서 설명, 면접 단골)** — *BetSettled consume 멱등·상태전이 가드*: Kafka at-least-once라 재배달됨. 이미 SETTLED면 no-op(멱등, betId 기준), unknown betId/불법 전이(ACCEPTED 아님)는 throw → DLT(조용한 손상 대신 운영 신호). maxPayout(수락 시 최악, 감사)과 settledPayout(실제 지급)을 V3에서 분리. Avro는 edge에서만 다뤄 wire schema가 aggregate에 안 샘.
- **L2 (문서 보며 설명)** — V3 check 제약(결과 유효/payout 쌍/비음수/currency match), autoStartup 주차로 비Kafka 테스트 격리, DLT 라우팅이 retry 소진 후.
- **L1 (읽으면 됨)** — SETTLED 컬럼 구성, topic `bet.settled.v1`, LOST면 payout 0.
