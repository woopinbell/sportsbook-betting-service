# 011 — risk/wallet 응답 변환 WireMock 테스트 (`test(client): WireMock coverage for risk/wallet response translation`)

## 개요 — 왜 이 변경이 필요한가

010의 핵심은 **응답 변환**(200 approved:false → 비즈니스, 5xx/timeout → 인프라)이다. WireMock으로 각 의존을 stub해 **메서드 본문의 변환**을 breaker와 독립적으로 검증한다.

## 작업 순서

1. `RiskClientTest` — approved / declined / 5xx / read-timeout.
2. `WalletClientTest` — 200 / 422 / 5xx / read-timeout / refund.

## 작업 내역

WireMock은 실제 HTTP 서버를 띄워 stub 응답을 준다 — 매핑이 진짜 와이어 위에서 동작함을 본다(mock 객체보다 충실).

- **RiskClient**: approved → 정상 반환; `approved:false` → `RiskLimitException`; 5xx·read-timeout → `DependencyUnavailableException`.
- **WalletClient**: 200 → operation id 반환(그리고 **betId가 Idempotency-Key 헤더로** 나감을 확인); 422 `WALLET_INSUFFICIENT_BALANCE` → `InsufficientBalanceException`; 5xx·read-timeout → `DependencyUnavailableException`; refund가 `/credit`에 `refund:<betId>` 키로 POST됨.

이 테스트들은 **AOP breaker와 독립**이다 — 변환이 메서드 본문에 있으므로(010), proxy 없이도 매핑이 검증된다.

> **C/C++ 관점**: WireMock은 테스트 동안 로컬 포트에 가짜 HTTP 서버를 띄운다(통합 mock). read-timeout 케이스는 stub이 응답을 일부러 지연시켜 client의 500ms read timeout이 `DependencyUnavailableException`으로 떨어지는지 본다 — 네트워크 실패를 결정적으로 재현.

## 결과

010의 모든 변환 분기가 실 HTTP 위에서 검증됨.

## 요약

- WireMock stub으로 approved/declined/5xx/timeout/422/refund 변환 검증.
- betId가 debit Idempotency-Key 헤더로 나감, refund는 `refund:<betId>` 키로 `/credit`.
- breaker와 독립(본문 변환).
