# 010 — 동기 risk + wallet 클라이언트 (Resilience4j) (`feat(client): add synchronous risk + wallet clients with Resilience4j`)

## 개요 — 왜 이 변경이 필요한가

ADR-0017의 접수 critical path에서 betting이 조율하는 두 HTTP 의존이다. 각각 `RestClient` + fail-fast 타임아웃 + circuit breaker. **핵심 설계점은 비즈니스 거절과 인프라 실패를 분리**하는 것 — 둘은 완전히 다르게 다뤄야 한다(fail-closed는 후자에만).

## 작업 순서

1. `ClientConfig`/`ClientProperties` — 엔드포인트·타임아웃 바인딩, 두 `RestClient`.
2. `RiskClient` — approved:false → 비즈니스 예외.
3. `WalletClient` — 422 → 비즈니스 예외, betId 멱등 키, refund.
4. Resilience4j breaker(인프라 예외만 record).

## 작업 내역

### 두 종류의 실패

```
비즈니스 거절(정상 판정):  risk approved:false (HTTP 200!),  wallet 422 INSUFFICIENT_BALANCE
인프라 실패(의존이 아픔):  timeout / connection / 5xx / 예상 못한 응답
```

- **RiskClient**: risk는 거절도 **HTTP 200**으로 답한다 — `approved:false`를 `RiskLimitException`(LIMIT_EXCEEDED)으로 변환. transport 실패만 throw.
- **WalletClient**: wallet은 422 `code=WALLET_INSUFFICIENT_BALANCE` 하나만 비즈니스 거절 → `InsufficientBalanceException`(shared `INSUFFICIENT_BALANCE`/409 재표현). **betId가 debit의 Idempotency-Key**. roll-back 환불은 `USER_LOCKED`에서 `refund:<betId>`라는 *별도* 키로 credit.

타임아웃/연결/5xx/예상 못한 응답은 전부 `DependencyUnavailableException`(INTERNAL_ERROR)로 — **메서드 본문 안에서** 변환하므로 AOP proxy 유무와 무관하게 성립한다.

### circuit breaker는 인프라 예외만 센다

```yaml
record-exceptions:
  - com.sportsbook.betting.error.DependencyUnavailableException   # 이것만
```

breaker가 record하는 건 `DependencyUnavailableException` 하나뿐이다. 그래서 **비즈니스 거절은 breaker를 절대 트립하지 않는다** — 잔고 부족 사용자가 몰려도 breaker가 열려 멀쩡한 호출을 fallback으로 빼는 일이 없다. fallback은 회로가 열렸을 때(`CallNotPermittedException`)만 발동하고, 비즈니스 예외는 그대로 rethrow한다.

> **C/C++ 관점**: Resilience4j `@CircuitBreaker`는 Spring AOP proxy로 메서드를 감싼다. proxy가 메서드 *반환/예외*를 관찰해 회로 상태를 갱신한다. 변환을 메서드 *본문*에서 미리 해두면 proxy가 없는 단위 테스트(011)에서도 같은 매핑이 성립 — proxy에 의존하지 않는 방어적 설계.

### V1 ErrorCode 한계

`DependencyUnavailableException`이 INTERNAL_ERROR(500)로 매핑된다 — shared ErrorCode 카탈로그에 V1 `SERVICE_UNAVAILABLE`(503)가 없어서다. 의미상 503이 맞지만 V1 한계로 문서화(retrospective "남은 한계").

## 결과

risk/wallet 동기 호출 + fail-fast + breaker. 비즈니스/인프라 실패가 타입으로 갈림.

## 요약

- 두 의존을 `RestClient` + 타임아웃 + breaker로.
- **비즈니스 거절(risk 200 approved:false, wallet 422) vs 인프라 실패(timeout/5xx)** 를 타입 분리.
- breaker는 `DependencyUnavailableException`만 record → 거절은 절대 트립 안 함. betId = debit 멱등 키, refund는 `refund:<betId>` 별도 키.

## 기억/설명 Level

- **L3 (외워서 설명, 면접 단골)** — *circuit breaker가 비즈니스 거절을 세면 안 되는 이유*: risk는 거절도 HTTP 200, wallet은 422가 비즈니스. 이걸 인프라 실패(timeout/5xx)와 타입으로 갈라, breaker의 record-exceptions를 `DependencyUnavailableException`만으로 좁힌다. 안 그러면 잔고 부족 트래픽이 회로를 열어 멀쩡한 의존을 fallback으로 뺀다. betId를 wallet debit 멱등 키로 전파해 재시도/reconcile 안전.
- **L2 (문서 보며 설명)** — 변환을 메서드 본문에서 하는 이유(proxy 무관), fallback은 open circuit에서만, refund의 별도 키(`refund:<betId>`).
- **L1 (읽으면 됨)** — connect 200ms/read 500ms 타임아웃, 503 부재로 500 매핑(V1 한계).
