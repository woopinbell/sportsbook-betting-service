# 018 — reconciliation (stale PENDING 보상) (`feat(reconciliation): resolve stale PENDING bets (ADR-0017 compensation)`)

## 개요 — 왜 이 변경이 필요한가

014가 남긴 분산 일관성 구간을 닫는다: **wallet debit은 성공했는데 accept 트랜잭션이 유실**(프로세스 다운/DB blip)되면 bet이 PENDING으로 박힌다 — 돈은 잡혔는데 베팅은 미확정. `BetReconciliationJob`이 오래된 PENDING을 멱등 re-debit으로 roll-forward 또는 roll-back한다(ADR-0017 보상).

## 작업 순서

1. `BetReconciliationJob` — stale PENDING 스캔 + 건별 resolve.
2. roll-forward / roll-back / defer 분기.
3. `BetStore` 메서드를 PENDING 가드로 멱등화.

## 작업 내역

### re-debit이 상태 probe이자 roll-forward를 겸한다

```java
private void reconcileOne(Bet bet) {
  Money totalStake = calculator.totalStake(bet.slipType(), bet.stake(), bet.legs().size());
  try {
    walletClient.debit(bet.betId(), bet.userId(), totalStake);  // 멱등 re-debit (같은 betId 키)
    store.acceptAndEnqueue(...);   // debit ok → roll forward ACCEPTED + outbox
  } catch (InsufficientBalanceException notDebited) {
    store.markRejected(bet.betId(), "RECONCILED_NO_DEBIT", clock.instant());  // roll back
  } catch (DependencyUnavailableException unavailable) {
    // wallet 불가 → 그대로 PENDING, 다음 tick (defer)
  }
}
```

wallet에 "키로 조회" 엔드포인트가 없고 debit이 멱등(betId 키)이라, **re-debit이 곧 상태 조회 + roll-forward**다:
- **debit 200** → 돈이 (지금 또는 이전에) 잡혀 있음 → roll-forward(ACCEPTED + outbox emit),
- **422 insufficient** → 차감이 애초에 없었고 될 수도 없음 → roll-back(REJECTED, reason `RECONCILED_NO_DEBIT`),
- **503** → wallet 불가 → 그대로 PENDING, 다음 tick으로.

### roll-back에 보상 credit을 넣으면 안 된다 (핵심 함정)

직관은 "취소하니 환불"이지만, **422가 떴다는 건 차감이 애초에 안 일어났다**는 뜻이다. 게다가 wallet의 `locked` 버킷은 **per-account이지 per-bet이 아니다** — 차감된 적 없는 stake를 환불하면 **다른 베팅이 잡아둔 돈을 끌어다 쓰는** 버그가 된다. 그래서 roll-back은 REJECTED 표시만 하고, 환불 primitive(`WalletClient.refund`)는 *확정 차감 후 포기* 흐름(정산/admin)에만 예약한다.

### tick 멱등화

reconciliation은 주기적이라 같은 bet을 여러 번 볼 수 있다. `BetStore.acceptAndEnqueue`/`markRejected`를 **PENDING 가드**로 만들어(이미 PENDING 아니면 no-op), 반복 tick이 중복 outbox 행을 만들거나 이미 정착한 상태를 덮지 않게 한다.

> **C/C++ 관점**: 이건 분산 시스템의 "보상 트랜잭션(compensating transaction)"이다. 2PC(분산 락) 대신, 각 연산을 **멱등**하게 만들고 *나중에* 불일치를 수렴시킨다. 멱등성(betId 키) 덕에 re-debit/accept를 몇 번 재시도해도 안전 — eventual consistency를 안전하게 다루는 표준 패턴.

## 결과

부분 실패 구간(PENDING+차감)이 닫힘. 돈을 절대 잃지 않고(roll-forward), 차감 안 된 건 깔끔히 거절(roll-back), wallet 장애는 미룸(defer). 반복 tick 안전.

## 요약

- 멱등 re-debit이 **상태 probe + roll-forward 겸용**(wallet 키 조회 엔드포인트 부재 우회).
- **roll-back에 환불 금지** — 422 = 차감 안 됨 + locked 버킷이 per-account라 남의 돈 끌어옴.
- `BetStore` 메서드 PENDING 가드로 tick 멱등(중복 outbox 방지).

## 기억/설명 Level

- **L3 (외워서 설명, 면접 단골)** — *reconciliation 보상*: wallet debit 성공 후 accept 유실 → PENDING+차감 구간. 잡이 stale PENDING을 멱등 re-debit(같은 betId 키). 200=roll-forward(ACCEPTED), 422=roll-back(REJECTED), 503=defer. **roll-back에 환불 안 함** — 422면 차감이 애초에 없었고, wallet locked는 per-account라 환불하면 남의 베팅 held 자금을 끌어옴. tick 멱등(PENDING 가드).
- **L2 (문서 보며 설명)** — re-debit이 키 조회 부재를 겸하는 이유, 보상 트랜잭션 = 2PC 대신 멱등+수렴, refund는 확정 차감 후 포기 흐름에만.
- **L1 (읽으면 됨)** — pending-timeout 30s, reason `RECONCILED_NO_DEBIT`.
