# 003 — V1 스키마 (bet + bet_leg) (`build(flyway): add V1__bet_and_leg schema`)

## 개요 — 왜 이 변경이 필요한가

001의 STI aggregate를 받칠 DB 스키마. Flyway V1으로 `bet` + `bet_leg` 테이블을 만들고, 멱등성·구조·시간 정합을 DB 제약으로 박는다. JPA 매핑이 스키마와 일치하는지 Testcontainers로 증명한다(ddl-auto:validate라 불일치면 컨텍스트 부팅 실패).

## 작업 순서

1. `V1__bet_and_leg.sql` — `bet` + `bet_leg` + 제약 + 인덱스.
2. `BetRepository`에 멱등 lookup + `@EntityGraph` leg fetch.
3. `BetPersistenceIntegrationTest`(Testcontainers PostgreSQL)로 매핑·제약 증명.

## 작업 내역

### 핵심 제약

```sql
CONSTRAINT uk_bet_idempotency_key UNIQUE (idempotency_key),   -- ADR-0005 강한 멱등 보증
CONSTRAINT uk_bet_reference       UNIQUE (bet_reference),     -- 사람용 B-YYYY-MM-DD-XXXX (ADR-0003)
CONSTRAINT bet_currency_match     CHECK (stake_currency = max_payout_currency),
CONSTRAINT bet_system_params CHECK (   -- K-of-N 파라미터는 SYSTEM 슬립에만 존재
    (slip_type =  'SYSTEM' AND system_min_wins IS NOT NULL AND system_total_selections IS NOT NULL)
 OR (slip_type <> 'SYSTEM' AND system_min_wins IS     NULL AND system_total_selections IS     NULL))
```

`uk_bet_idempotency_key`가 ADR-0005의 **강한 멱등 보증**이다 — 같은 Idempotency-Key로 재요청하면 여기서 충돌하고 같은 betId를 replay한다(014에서 활용). 시간 컬럼은 전부 `TIMESTAMP WITH TIME ZONE`(ADR-0003 Instant/UTC).

### 두 인덱스의 의도

```sql
CREATE INDEX ix_bet_user_bet ON bet (user_id, bet_id DESC);     -- cursor pagination
CREATE INDEX ix_bet_status_created ON bet (status, created_at); -- reconciliation 스캔
```

`bet_id`가 UUID v7(시간순)이라 cursor 그 자체로 쓸 수 있다(020). `ix_bet_status_created`는 reconciliation의 "PENDING older than T" 스캔용(018).

> **C/C++ 관점**: `NUMERIC(9,4)`(bet_leg odds)는 부동소수점이 아니라 **고정 소수**다. 돈·odds에 `double`을 쓰지 않는 이유와 같다 — 0.1을 정확히 표현 못 하는 IEEE 754의 반올림 누적을 피한다. Java 쪽은 `BigDecimal`로 대응(`Money`는 minor unit `long`).

### ddl-auto:validate — 매핑이 스키마와 일치하는지 부팅이 검증

`BetPersistenceIntegrationTest`가 Testcontainers PostgreSQL을 띄워 SYSTEM 슬립 + leg를 왕복하고, 두 UNIQUE 제약이 중복을 거절함을 단언한다. Hibernate가 `validate` 모드라 매핑이 스키마와 어긋나면 **컨텍스트 부팅 자체가 실패**한다 — 즉 테스트가 도는 것 자체가 매핑 일치의 증거.

## 결과

스키마 + JPA 매핑 정합 증명. SYSTEM 슬립 영속·복원, 멱등/reference 중복 거절 확인.

## 요약

- `bet` + `bet_leg` 스키마, 멱등(`uk_bet_idempotency_key`)·구조(`bet_system_params`)·시간(timestamptz) 제약.
- cursor용/reconciliation용 인덱스 2개.
- ddl-auto:validate + Testcontainers로 매핑 일치를 부팅이 보증.

## 기억/설명 Level

- **L3 (외워서 설명)** — *멱등의 최종 권위는 DB*: `uk_bet_idempotency_key` UNIQUE가 강한 보증, Redis(013/014)는 가속일 뿐. `bet_system_params` CHECK로 K-of-N 파라미터가 SYSTEM에만 존재함을 DB가 강제.
- **L2 (문서 보며 설명)** — ddl-auto:validate가 매핑 불일치를 부팅에서 잡는다, UUID v7이 cursor 겸용, ix_bet_status_created가 reconciliation 스캔용.
- **L1 (읽으면 됨)** — NUMERIC(9,4) 고정소수, timestamptz, 인덱스 컬럼 순서.
