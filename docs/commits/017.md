# 017 — publisher → embedded Kafka 검증 (`test(outbox): publisher drains to embedded Kafka`)

## 개요 — 왜 이 변경이 필요한가

016의 publisher가 **실제 브로커**로 행을 내보내는지, 그리고 record가 올바른 partition key + 디코드 가능한 Avro payload를 갖는지 검증한다. `@EmbeddedKafka`로 진짜 브로커를 띄운다.

## 작업 순서

1. `@EmbeddedKafka` + Testcontainers PostgreSQL/Redis.
2. outbox 행 저장 → publisher 구동 → 브로커에서 consume → 단언.
3. empty 시 no-op 케이스.

## 작업 내역

`BetPlacedRequested` outbox 행을 저장하고 publisher를 명시적으로 돌린 뒤(016에서 테스트는 주차됨), 브로커에서 record를 consume해 단언한다:
- record가 **user partition key**를 운반(013의 결정),
- Avro payload가 같은 betId로 디코드됨,
- `published_at`이 stamp됨.

plus **empty면 no-op**(미발행 행이 없을 때 아무것도 안 함).

> **C/C++ 관점**: `@EmbeddedKafka`는 테스트 JVM 안에 인메모리 Kafka 브로커를 띄운다(통합). Testcontainers(별도 컨테이너)와 달리 같은 프로세스라 빠르지만, 실제 Kafka 프로토콜로 produce/consume을 검증한다 — mock이 아니라 진짜 브로커 동작.

## 결과

publisher가 실 브로커로 drain하고 partition key·payload·stamp가 맞음을 증명. 016의 at-least-once 골격이 실제로 동작.

## 요약

- `@EmbeddedKafka`로 실 브로커 drain 검증: user partition key, Avro betId 디코드, published_at stamp.
- empty no-op 케이스.
