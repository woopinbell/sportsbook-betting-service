# 004 — 영속 테스트를 JPA slice로 (`refactor(test): scope Bet persistence test to a JPA slice`)

## 개요 — 왜 이 변경이 필요한가

003의 `BetPersistenceIntegrationTest`는 처음엔 전체 `@SpringBootTest`였다. 매핑 테스트는 DB만 있으면 되는데, 풀 컨텍스트는 곧 추가될 Redis(slippage)·Kafka(outbox)·HTTP 클라이언트 빈까지 끌어와 **이 테스트가 쓰지도 않는 인프라를 띄우게** 만든다. slice로 좁혀 결합을 끊는다.

## 작업 순서

1. `@SpringBootTest` → `@DataJpaTest`(Testcontainers PostgreSQL, `replace = NONE`, `ddl-auto = validate`).
2. 왕복에 `flush()` + `clear()`를 넣어 진짜 DB read를 강제.

## 작업 내역

`@DataJpaTest`는 **JPA slice** — 컴포넌트 그래프의 나머지(서비스·클라이언트·리스너)를 로드하지 않고 repository·EntityManager·DataSource만 띄운다. `replace = NONE`은 Spring이 임베디드 DB로 바꿔치기하지 못하게 해 Testcontainers PostgreSQL을 실제로 쓰게 한다.

왕복에서 `EntityManager`를 flush + clear하는 게 중요하다. clear 없이 find하면 **1차 캐시(persistence context)** 에 남은 같은 인스턴스가 돌아와 "DB에서 진짜 읽혔는지"를 검증하지 못한다. clear로 캐시를 비워야 SELECT가 실제로 나간다.

> **C/C++ 관점**: JPA의 persistence context는 **identity map 캐시**다. 같은 트랜잭션에서 같은 PK를 두 번 find하면 두 번째는 DB가 아니라 메모리에서 온다(같은 객체 참조). C++에서 객체 포인터를 캐싱해 두는 것과 비슷하지만, JPA는 이걸 자동으로 한다 — 그래서 "DB read 검증"엔 명시적 clear가 필요.

## 결과

매핑 테스트가 DB만 띄우는 가벼운 slice가 됨. 이후 Redis/Kafka/HTTP 빈이 늘어도 이 테스트를 부풀리지 않는다.

## 요약

- `@SpringBootTest` → `@DataJpaTest` slice(`replace = NONE`로 Testcontainers 유지).
- flush + clear로 1차 캐시를 비워 진짜 DB read 검증.
