# 006 — odds slippage checker (odds-feed 캐시) (`feat(validation): add odds slippage checker over the odds-feed cache`)

## 개요 — 왜 이 변경이 필요한가

odds는 초 단위로 변한다(ADR-0017의 동기 접수 근거). 사용자가 본 odds와 접수 시점 odds가 어긋날 수 있어, 접수 전 **현재 odds**와 **제출 odds**를 3% tolerance로 비교해야 한다(ADR-0008). odds 권위는 odds-feed-service이고, betting은 그 Redis 캐시의 **read-only 소비자**다(키를 절대 쓰지 않는다).

## 작업 순서

1. `OddsSlippageChecker` — market 상태 + 가격 + tolerance 체크.
2. cross-multiplication으로 division 반올림 회피.

## 작업 내역

### 두 Redis 키, read-only

```
market:{eventId}:{marketId}                 -> 상태(OPEN 이어야 함)
odds:{eventId}:{marketId}:{selectionId}     -> 현재 decimal odds(string)
```

market 상태가 `OPEN`이 아니면 `MarketClosedException`(EVENT_CLOSED) — suspended/closed/missing 모두 포괄. selection이 더 이상 가격이 없으면 같은 예외. betting은 이 키를 **읽기만** 한다 — 쓰기는 odds-feed의 책임(경계 분리).

### 사용자 보호적 규칙

drift 규칙은 **사용자에게 유리한 방향은 항상 통과**다. 현재 odds가 `submitted × (1 - tolerance)` 이상이면 수락 — 즉 odds가 *올라간*(개선된) 건 무조건 OK, *tolerance 넘게 떨어진* 것만 `OddsDriftException`(ODDS_DRIFT)으로 거절.

### division 없이 비교

```java
// current >= submitted * (100 - tolerance) / 100  를 나눗셈 없이:
//   current * 100 >= submitted * (100 - tolerance)
BigDecimal currentScaled = current.multiply(HUNDRED);
BigDecimal floor = submitted.multiply(toleranceFactor);
if (currentScaled.compareTo(floor) < 0) throw new OddsDriftException(...);
```

나눗셈을 쓰면 `BigDecimal`이 무한소수에서 scale/rounding을 강요해 경계가 흐려진다. **교차 곱(cross-multiplication)** 으로 정수 비교처럼 만들어 경계를 정확히. 003의 odds=2.0000이면 3% floor = 1.9400 — 정확히 그 값에서 통과, 1.9399에서 거절(007 테스트가 이 경계를 박는다).

> **C/C++ 관점**: `a/b >= c/d`를 `a*d >= c*b`로 바꾸는 건 정수 산술에서 익숙한 트릭이다. `BigDecimal`은 임의정밀이지만 나눗셈만은 정밀도가 무한일 수 있어 명시적 scale가 필요 — 그래서 가능하면 곱셈으로 우회.

## 결과

market 상태·가격·tolerance 검증. 접수 경로의 두 번째 게이트(외부 HTTP 전, Redis read만).

## 요약

- odds-feed Redis 캐시의 **read-only** 소비자(`market:*`, `odds:*` 키).
- market이 OPEN 아니면 EVENT_CLOSED, tolerance(3%) 넘는 *하락*만 ODDS_DRIFT(개선은 통과).
- 비교를 cross-multiplication으로 — division 반올림 회피, 경계 정확.

## 기억/설명 Level

- **L3 (외워서 설명)** — *slippage가 사용자 보호적인 이유 + 경계 계산*: 현재 odds ≥ 제출×(1−3%)면 수락, odds 개선은 항상 통과, tolerance 넘는 하락만 거절. 경계는 cross-multiplication(`current*100 ≥ submitted*(100−tol)`)으로 division 반올림 없이. 권위는 odds-feed, betting은 read-only.
- **L2 (문서 보며 설명)** — 두 Redis 키 형식, market 상태가 OPEN 아니면(suspended/closed/missing) EVENT_CLOSED.
- **L1 (읽으면 됨)** — tolerance 기본 3%, 키 prefix `market:`/`odds:`.
