# 001 — Bet aggregate + BetLeg (STI 슬립 모델) (`feat(domain): introduce Bet aggregate + BetLeg entity (STI slip model)`)

## 개요 — 왜 이 변경이 필요한가

베팅 서비스의 모든 것이 슬립(slip) 위에 선다. 슬립 타입이 Single / Multiple / **System(K-of-N)** 세 가지인데, 이를 어떻게 한 모델로 표현하느냐가 첫 도메인 결정이다. ADR-0008은 **STI(Single Table Inheritance)** — `bet` 한 테이블 + `slip_type` 판별 컬럼 — 를 박았다. 이 커밋은 그 aggregate root(`Bet`)와 자식 엔티티(`BetLeg`)를 만든다.

## 작업 순서

1. `SlipKind` enum(SINGLE/MULTIPLE/SYSTEM) — shared-protocol의 rich `BetSlipType` sealed interface를 영속용 평면 표현으로.
2. `Bet` aggregate — STI 컬럼 + 불변식 검증 + 생애주기 전이.
3. `BetLeg` child — selection 1개 = 1행, 제출 순서 보존.
4. `EmbeddedMoney`(JPA용 Money 미러) + `UuidV7`(시간순 PK).

## 작업 내역

### rich 타입을 평면 컬럼으로, 다시 복원

shared-protocol의 `BetSlipType`은 sealed interface(`Single`/`Multiple`/`System(minWins, totalSelections)`)다. JPA는 이런 cross-module sealed 타입을 그대로 영속하기 어렵다. 그래서 평면 `SlipKind` enum + nullable `system_min_wins`/`system_total_selections` 컬럼으로 분해하고, `slipType()` 메서드로 rich 타입을 복원한다. K-of-N 파라미터는 SYSTEM 슬립에만 존재.

```java
public BetSlipType slipType() {
  return switch (slipType) {
    case SINGLE   -> new BetSlipType.Single();
    case MULTIPLE -> new BetSlipType.Multiple();
    case SYSTEM   -> new BetSlipType.System(systemMinWins, systemTotalSelections);
  };
}
```

> **C/C++ 관점**: Java의 `sealed interface` + record는 C의 tagged union(`enum tag + union payload`)과 같은 의도다. `switch`가 모든 case를 강제하므로(pattern matching), 새 슬립 타입을 추가하면 컴파일러가 빠진 분기를 잡아준다.

### 불변식: aggregate가 절대 깨면 안 되는 것

생성 팩토리 `Bet.pending(...)`가 강제하는 건 **항상 참인 불변식**이다:
- stake와 maxPayout의 **currency 일치**(슬립 내 FX 없음, V1),
- **구조적 형태**: SINGLE = 1 leg, MULTIPLE ≥ 2 legs, SYSTEM totalSelections = leg 수.

L1/L2/L4/L5 같은 **정책 규칙**은 별도 validator(005)의 몫이다. 이 둘을 섞지 않은 게 핵심 — 정책은 설정으로 바뀔 수 있지만 불변식은 코드 정합성이다.

### 생애주기는 이 서비스가 소유한 전이만

`Bet`이 노출하는 전이는 접수 단계 것뿐: `PENDING → ACCEPTED`(accept) / `PENDING → REJECTED`(reject). SETTLED/CANCELLED/VOIDED는 다른 서비스 소관이라 이 커밋엔 없다(SETTLED/VOIDED는 Phase 4의 024~026에서 추가). 재수락이나 수락 후 거절은 `IllegalStateException`.

### maxPayout은 입력, 계산은 도메인 서비스로

`maxPayout`을 엔티티 안에서 계산하지 않고 **생성자 입력**으로 받는다. K-of-N 전개·payout 계산은 별도 도메인 서비스(`SystemBetCalculator`, 008)가 한다. 엔티티는 pricing 로직에서 자유롭게.

### EmbeddedMoney와 UuidV7

`EmbeddedMoney`는 shared-protocol `Money` record를 JPA `@Embeddable`로 미러한다 — Hibernate가 cross-module record를 embed 못 하므로 **각 서비스가 자기 영속 타입을 소유**한다. `UuidV7`(RFC 9562)은 bet/leg PK에 시간순 prefix를 줘 B-tree insert가 친화적(ADR-0003) — wallet-service와 같은 생성기.

## 결과

`Bet`/`BetLeg` 엔티티 + 지원 타입. 아직 스키마(003)·검증(005)·계산(008)이 없어 컴파일·단위 테스트만.

## 요약

- 슬립을 **STI**로 모델링(ADR-0008): `bet` + `slip_type` + nullable K-of-N 컬럼.
- rich `BetSlipType` ↔ 평면 `SlipKind`+컬럼 왕복.
- **불변식**(currency·구조)은 aggregate, **정책**(L1/L2/L4/L5)은 별도 validator로 분리.
- 생애주기는 접수 전이(PENDING→ACCEPTED/REJECTED)만 소유.

## 기억/설명 Level

- **L3 (외워서 설명)** — *왜 STI인가 / 불변식 vs 정책 분리*: 슬립 3타입을 한 테이블 + 판별 컬럼으로(ADR-0008), K-of-N은 SYSTEM에만. aggregate는 *항상 참*인 불변식(currency·구조)만 강제하고, 설정으로 바뀌는 정책(L1/L2/L4/L5)은 validator로 분리.
- **L2 (문서 보며 설명)** — sealed `BetSlipType` ↔ `SlipKind`+컬럼 복원, maxPayout을 입력으로 받아 엔티티를 pricing에서 분리, EmbeddedMoney(서비스별 JPA 타입 소유), UuidV7 시간순 PK 이유.
- **L1 (읽으면 됨)** — leg가 제출 순서로 인덱스 부여, 생애주기 전이 메서드 이름.
