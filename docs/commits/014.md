# 014 — 동기 접수 orchestration (ADR-0017) (`feat(placement): synchronous bet-placement orchestration (ADR-0017)`)

## 개요 — 왜 이 변경이 필요한가

이 repo의 심장. ADR-0017의 동기 접수 흐름을 조립한다: 멱등 replay → 슬립+stake 검증 → odds slippage → PENDING 기록 → risk check(HTTP) → wallet debit(HTTP) → accept + outbox. **트랜잭션 경계**(외부 HTTP를 DB 트랜잭션에 넣지 않기)가 이 커밋의 진짜 결정이다.

## 작업 순서

1. `PlaceBetCommand` — 접수 입력.
2. `BetStore` — 짧은 트랜잭션 단위(별도 빈).
3. `IdempotencyCache` — Redis SETNX 빠른 경로.
4. `BetPlacementService` — orchestrator(비트랜잭션).

## 작업 내역

### 트랜잭션 경계가 핵심 — orchestrator는 @Transactional 아님

```java
// BetPlacementService.place() — NOT @Transactional
store.savePending(bet);                       // 짧은 트랜잭션 #1 (즉시 커밋)
riskClient.check(...);                         // HTTP — 트랜잭션 밖
walletClient.debit(...);                       // HTTP — 트랜잭션 밖
Bet accepted = store.acceptAndEnqueue(betId, event, now);  // 짧은 트랜잭션 #2 (accept+outbox 원자)
```

만약 `place()` 전체에 `@Transactional`을 걸면, risk/wallet HTTP 왕복(수백 ms) 동안 **풀링된 DB 커넥션을 점유**한다 — 1만 동시 베팅이면 HikariCP 20개 풀이 즉시 고갈되고 느린 wallet 하나가 전체 접수를 마비시킨다. 그래서 orchestrator를 **비트랜잭션**으로 두고, DB 단계를 별도 빈 `BetStore`의 짧은 트랜잭션으로 쪼개 HTTP 호출을 *트랜잭션 사이*에 둔다.

`BetStore`를 별도 빈으로 뺀 두 번째 이유: `@Transactional`은 Spring AOP proxy로 동작하므로 **self-invocation(같은 빈 내부 호출)이면 트랜잭션이 안 걸린다**. 별도 빈이라야 proxy를 통과한다.

> **C/C++ 관점**: 이게 Spring 트랜잭션의 가장 흔한 함정이다. 메서드에 `@Transactional`을 붙였다고 항상 트랜잭션이 생기는 게 아니라, **프록시 경계를 넘는 호출**에만 적용된다. 같은 객체의 `this.method()`는 프록시를 우회해 애너테이션이 무시된다. C++에서 데코레이터로 감싼 객체를 우회해 raw `this`를 부르는 것과 같은 문제.

### 멱등성 2겹 + validate를 먼저

```java
1. store.findByIdempotencyKey(key)  // DB가 권위 — 이미 커밋된 bet이면 반환
2-3. validate + slippage            // side effect 전 — 잘못된 슬립은 키/행을 안 태움
4. idempotency.tryReserve(key)      // Redis SETNX 빠른 경로
   store.savePending(bet)           // saveAndFlush → uk_bet_idempotency_key 충돌이 여기서 즉시
```

- **Redis SETNX**(`IdempotencyCache`)는 빠른 경로(이미 처리 중이면 일찍 단락), **DB unique**(`uk_bet_idempotency_key`)가 강한 보증(ADR-0005). 동시 승자가 생기면 `savePending`(saveAndFlush)이 `DataIntegrityViolationException`을 *그 자리에서* 던지고, 잡아서 커밋된 bet을 반환한다.
- **validate를 reservation 앞에** 둬서 잘못된 슬립은 키도 행도 태우지 않는다.

### betId를 멱등 키로, 거절은 즉시 REJECTED

```java
UUID betId = UuidV7.generate();   // 미리 생성
...
try {
  riskClient.check(betId, ...);
  walletClient.debit(betId, ...);   // betId = wallet 멱등 키 → 재시도/reconcile 안전
} catch (BetPlacementException rejection) {
  store.markRejected(betId, rejection.errorCode().name(), clock.instant());  // PENDING → REJECTED
  throw rejection;   // RFC 7807 매핑으로
}
```

`betId`를 *미리* 생성해 wallet debit의 멱등 키로 쓴다(018 reconciliation이 같은 키로 re-debit). risk/wallet 거절이면 PENDING 행을 REJECTED로 뒤집고(ErrorCode 이름 저장) 예외를 rethrow해 컨트롤러가 RFC 7807로 렌더한다. UTC `Clock` 빈으로 모든 Instant가 timezone-safe.

## 결과

ADR-0017 동기 접수 흐름이 동작. DB 커넥션이 HTTP 왕복을 절대 가로지르지 않고, 멱등 2겹이 중복 수락을 막는다. 단, savePending 후 accept 전 프로세스 다운 시 PENDING+차감됨 구간이 남는다 → 018 reconciliation이 닫는다.

## 요약

- **orchestrator는 비트랜잭션** — DB 단계를 `BetStore` 짧은 트랜잭션으로 쪼개고 HTTP를 그 사이에. 커넥션이 네트워크 왕복을 가로지르지 않음(ADR-0017 핵심).
- `BetStore` 별도 빈 = proxy 통과(self-invocation 함정 회피) + 경계 강제.
- 멱등 2겹(Redis SETNX 가속 + DB unique 권위), validate를 reservation 앞에.
- betId 미리 생성 = wallet 멱등 키, 거절 즉시 REJECTED + rethrow.

## 기억/설명 Level

- **L3 (외워서 설명, 면접 최단골)** — *왜 외부 HTTP를 DB 트랜잭션에 안 넣나*: `place()` 전체에 `@Transactional`을 걸면 risk/wallet 왕복 동안 DB 커넥션을 점유 → 1만 동시면 풀 고갈, 느린 의존이 접수 마비. 그래서 orchestrator는 비트랜잭션, DB는 `BetStore` 짧은 트랜잭션으로 쪼개 HTTP를 사이에. `BetStore`를 별도 빈으로 뺀 건 (1) 경계 강제 (2) `@Transactional`이 프록시 기반이라 self-invocation이면 안 걸리는 함정 회피.
- **L3** — *멱등 2겹*: Redis SETNX(가속) + DB `uk_bet_idempotency_key`(권위). validate를 reservation 앞에 둬 잘못된 슬립이 키/행 안 태움. saveAndFlush가 중복을 즉시 터뜨려 커밋된 bet 반환. betId 미리 생성 = wallet 멱등 키(reconcile 안전).
- **L2 (문서 보며 설명)** — 거절 시 PENDING→REJECTED + rethrow, UTC Clock 빈, savePending 후 accept 전 다운이 PENDING+차감 구간을 남김(→018).
- **L1 (읽으면 됨)** — 7단계 흐름 순서.
