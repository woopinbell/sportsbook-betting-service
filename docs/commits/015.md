# 015 — 접수 흐름 통합 테스트 (`test(placement): integration coverage for the placement flow`)

## 개요 — 왜 이 변경이 필요한가

014의 동기 흐름은 **실제 인프라 위에서** 동작해야 한다 — PostgreSQL(영속), Redis(멱등/odds), HTTP stub(risk/wallet). 멱등·거절·정합 불변식을 풀 컨텍스트로 박는다. Kafka는 제외(outbox drain은 016).

## 작업 순서

1. `@SpringBootTest` + Testcontainers PostgreSQL/Redis + WireMock risk/wallet.
2. happy / risk decline / wallet 422 / 멱등 replay 케이스.

## 작업 내역

풀 `@SpringBootTest`라 014의 모든 빈이 실제로 배선된 채 돈다. Kafka만 제외(다음 작업).

- **happy path**: 수락 + max payout 계산 + **정확히 1개** `BetPlacedRequested` outbox 행(Avro → betId 디코드로 확인).
- **risk decline → REJECTED**: wallet은 **호출되기 전에** 거절, outbox 행 없음(순서가 risk→wallet이라 risk에서 끊김).
- **wallet 422 insufficient → REJECTED**: outbox 행 없음.
- **같은 Idempotency-Key 2회 → bet 1개, outbox 1개, wallet 정확히 1회 차감**: replay가 어떤 HTTP 호출 전에 단락(014 step 1).

happy의 outbox 행을 Avro로 디코드해 betId가 일치함을 확인하는 게 핵심 — 013의 직렬화와 014의 enqueue가 끝까지 맞물림을 증명.

## 결과

014의 동기 흐름·멱등·거절·정합이 실 인프라에서 증명됨. **단일 수락/단일 차감** 불변식의 권위 있는 증거(022 부하 테스트가 이를 인용).

## 요약

- 풀 컨텍스트(PostgreSQL/Redis/WireMock, Kafka 제외)로 happy/risk decline/wallet 422/멱등 replay 검증.
- risk decline은 wallet 호출 전, outbox 행은 happy에만 정확히 1개.
- 멱등 replay가 HTTP 전 단락 → wallet 1회 차감.
