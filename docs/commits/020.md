# 020 — REST 엔드포인트 + RFC 7807 (`feat(api): bet REST endpoints + RFC 7807 problem responses`)

## 개요 — 왜 이 변경이 필요한가

014의 접수 흐름을 HTTP로 노출한다. `BetController`가 내부 surface(`/internal/v1`, gateway가 공개 front door)를 제공하고, `BetExceptionHandler`가 모든 에러를 RFC 7807 `application/problem+json`으로 렌더한다(ADR-0004).

## 작업 순서

1. `BetController` — POST 접수 + GET by id + GET cursor list.
2. `BetExceptionHandler` — `@RestControllerAdvice`로 ErrorCode → problem+json.
3. eager leg fetch로 LazyInitialization 회피.

## 작업 내역

### 세 엔드포인트

```
POST /internal/v1/bets               Idempotency-Key 헤더 필수, 201 + Location
GET  /internal/v1/bets/{id}          단건, 없으면 404
GET  /internal/v1/bets?userId=&cursor=&limit=   newest-first, keyset pagination
```

- POST는 `Idempotency-Key` 헤더 필수 → DTO를 `PlaceBetCommand`로 매핑(slip type, odds, 멱등 키) → 014에 위임.
- list는 **keyset pagination**: 시간순 betId(UUID v7) 위로 cursor. `limit+1`을 fetch해 **count 쿼리 없이** `hasMore`를 결정한다.

> **C/C++ 관점**: keyset(cursor) pagination은 `WHERE bet_id < :cursor ORDER BY bet_id DESC LIMIT :n+1`로, OFFSET pagination(뒤로 갈수록 느려짐)과 달리 인덱스 위를 일정 비용으로 훑는다. `limit+1`을 가져와 n+1번째 존재 여부로 "다음 페이지 있음"을 판단 — count(*) 한 번을 아낀다.

### RFC 7807 — 하나의 advice가 ErrorCode를 렌더

```java
@ExceptionHandler(BetPlacementException.class)   // 하나로 모든 비즈니스 거절
public ResponseEntity<ProblemDetail> handlePlacement(BetPlacementException e, ...) {
  return problem(e.errorCode(), e.getMessage(), request);   // ErrorCode가 status/type/title 운반
}
```

`BetPlacementException`이 자기 `ErrorCode`를 들고 있어(005), **advice 하나가 모든 하위 타입을 처리**한다:
- ODDS_DRIFT 409 / LIMIT_EXCEEDED 403 / INSUFFICIENT_BALANCE 409 / EVENT_CLOSED 422 / DUPLICATE_BET 409 / fail-closed 500.

binding 실패 → VALIDATION_FAILED 400, 없는 bet → betting-local BET_NOT_FOUND 404. `correlationId`는 있으면 MDC trace id. shared `ErrorCode` 카탈로그를 재사용해 type URI/title/machine code가 서비스 간 일관.

### eager leg fetch — 트랜잭션 밖 렌더링

응답/조회 경로가 leg를 **eager**로 fetch한다(`@EntityGraph` on 멱등/cursor 쿼리, `BetStore.acceptAndEnqueue`). 슬립을 트랜잭션 닫힌 *후* 렌더하므로 `LazyInitializationException`(닫힌 세션에서 lazy 컬렉션 접근)을 피한다.

> **C/C++ 관점**: JPA의 lazy 로딩은 컬렉션을 처음 접근할 때 DB를 친다. 트랜잭션(세션)이 닫힌 뒤 접근하면 예외다 — `@EntityGraph`로 미리 JOIN FETCH해 두면 객체가 detach돼도 안전. C++에서 lazy 포인터를 dereference하기 전에 미리 로드해 두는 것과 같은 의도.

## 결과

접수/조회 HTTP surface + 일관된 RFC 7807 에러. gateway가 이 `/internal/v1`을 `/api/v1`로 프록시한다.

## 요약

- `/internal/v1/bets` POST(Idempotency-Key 필수, 201+Location) + GET by id + GET cursor list(`limit+1`로 count 없이 hasMore).
- `BetPlacementException`의 ErrorCode를 advice 하나가 RFC 7807로 렌더(shared 카탈로그 재사용).
- eager leg fetch로 트랜잭션 밖 렌더링 시 LazyInitialization 회피.

## 기억/설명 Level

- **L3 (외워서 설명)** — *RFC 7807 + ErrorCode*: 비즈니스 예외가 ErrorCode를 운반해 advice 하나가 모든 거절을 problem+json으로(ODDS_DRIFT 409/LIMIT_EXCEEDED 403/INSUFFICIENT_BALANCE 409/EVENT_CLOSED 422/DUPLICATE_BET 409). shared 카탈로그 재사용으로 서비스 간 일관. cursor pagination은 시간순 UUID v7 위 keyset, limit+1로 count 없이 hasMore.
- **L2 (문서 보며 설명)** — eager leg fetch가 LazyInitialization 회피, correlationId = MDC trace id, BET_NOT_FOUND는 betting-local.
- **L1 (읽으면 됨)** — 세 엔드포인트 시그니처, status code 매핑.
