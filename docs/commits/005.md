# 005 — 슬립 구조 validator + 정책 (`feat(validation): add slip structural validator with policy`)

## 개요 — 왜 이 변경이 필요한가

001에서 불변식과 분리한 **정책 규칙**(L1/L2/L4/L5)을 구현한다. 이건 설정(`application.yml`)으로 바뀔 수 있는 비즈니스 규칙이라 aggregate가 아니라 별도 validator가 소유한다(ADR-0008 / ADR-0009). 접수 경로에서 외부 호출 *전에* 돌아 잘못된 슬립을 싸게 거른다.

## 작업 순서

1. `BettingPolicyProperties`(`@ConfigurationProperties("betting.policy")`) — 타입 안전 바인딩 + 기본값.
2. `BetSlipValidator` — L4/L5/L1/L2 + stake 경계.
3. `BetPlacementException` 베이스 + `ValidationFailedException`(ErrorCode 운반).

## 작업 내역

### 정책은 타입 안전 바인딩, 누락 시 sane default

`BettingPolicyProperties`는 record로 `betting.policy.*`를 바인딩한다. compact 생성자가 ADR-0008/0009 기본값(max 15 selections, max odds 10000, 3% slippage, per-currency stake 경계)을 채운다 — 누락 키가 `0`/`null`로 떨어지지 않게. **잘못된 값은 request time이 아니라 startup에서 실패**한다.

> **C/C++ 관점**: Spring의 `@ConfigurationProperties`는 설정 파일을 강타입 객체로 바인딩한다 — C에서 ini/yaml을 손으로 파싱해 struct에 채우던 걸 프레임워크가 검증까지 해서 해준다. record의 compact 생성자(`public BettingPolicyProperties { ... }`)는 본문에서 필드를 정규화할 수 있는 자리다.

### 한 패스로 L1/L2, market을 먼저

```java
for (BetLeg leg : legs) {
  if (!markets.add(leg.marketId())) throw ... ("L1 same market ...");  // market 먼저
  if (!events.add(leg.eventId()))   throw ... ("L2 same event ...");
}
```

- **L4** max selections, **L5** max total odds(제출 odds의 곱),
- **L1**(Same Market 금지)·**L2**(Same Event 금지)는 **multi에만** 적용(single은 면제).

market 체크를 먼저 두는 게 의도적이다 — 같은 market을 공유하면 그건 동시에 같은 event이기도 하지만, **더 구체적인 L1**으로 보고해야 사용자에게 정확하다(L2로 뭉뚱그리지 않음).

### 비즈니스 거절 ≠ 인프라 실패

`ValidationFailedException`이 shared-protocol `ErrorCode`(`VALIDATION_FAILED`)를 운반한다. 새 베이스 `BetPlacementException`이 ErrorCode를 들고 있어, 컨트롤러 advice(020)가 하나의 RFC 7807 모양으로 렌더한다. 더 중요한 건 이게 **인프라 실패와 구분**된다는 점 — 010의 Resilience4j 배선에서 비즈니스 거절은 circuit breaker를 절대 트립하지 않아야 하므로 타입으로 갈라둔다.

## 결과

구조·stake 정책 검증 + ErrorCode 운반 예외 체계. 접수 경로의 첫 게이트.

## 요약

- 정책(L1/L2/L4/L5 + stake 경계)은 타입 안전 `BettingPolicyProperties`로 바인딩, 잘못된 값은 startup 실패.
- L1을 L2보다 먼저(더 구체적 보고), L1/L2는 multi에만.
- `BetPlacementException`이 ErrorCode 운반 → 인프라 실패와 타입 분리(010 breaker 배선 대비).

## 기억/설명 Level

- **L3 (외워서 설명)** — *불변식 vs 정책, 비즈니스 vs 인프라 예외*: L1/L2/L4/L5는 설정으로 바뀌는 정책이라 validator가 소유(aggregate 아님). 비즈니스 거절은 ErrorCode를 운반하는 `BetPlacementException`로, 인프라 실패와 타입을 갈라 circuit breaker가 거절을 안 세게 한다.
- **L2 (문서 보며 설명)** — `@ConfigurationProperties` 타입 안전 바인딩 + startup 검증, L1을 먼저 보고하는 이유(같은 market ⊂ 같은 event), single 면제.
- **L1 (읽으면 됨)** — max-selections 15, max-total-odds 10000, slippage 3% 기본값.
