# 021 — web-layer + problem-detail 테스트 (`test(api): web-layer + problem-detail coverage`)

## 개요 — 왜 이 변경이 필요한가

020의 HTTP 계약(상태 코드, problem+json 모양, errorCode 필드)을 **web slice**(`@WebMvcTest`)로 박는다. 서비스는 mock해 컨트롤러·advice만 검증.

## 작업 순서

1. `@WebMvcTest` + placement/query 서비스 mock.
2. 201 accept / 검증 실패 / 거절 매트릭스 / GET·list 케이스.

## 작업 내역

`@WebMvcTest`는 MVC 계층만 띄운다(서비스·DB·Redis 없음) — 컨트롤러 매핑과 예외 advice만 빠르게 검증.

- **201 accept**(Location 헤더 포함),
- **missing Idempotency-Key / empty selections → 400**,
- **거절 매트릭스 → RFC 7807**: LIMIT_EXCEEDED 403, INSUFFICIENT_BALANCE / ODDS_DRIFT / DUPLICATE_BET 409, EVENT_CLOSED 422, BET_NOT_FOUND 404 — 각각 status + `application/problem+json` content type + `errorCode` 필드를 단언.
- GET by id + cursor list 모양.

각 거절을 status·content-type·errorCode 세 축으로 단언해, 020의 advice가 ErrorCode를 빠짐없이 매핑함을 고정한다.

> **C/C++ 관점**: `@WebMvcTest`는 `MockMvc`로 실제 소켓 없이 HTTP 요청을 시뮬레이트한다(서블릿 디스패치만). 서비스를 `@MockBean`으로 갈아끼워 컨트롤러 로직만 격리 — 단위와 통합 사이의 slice 테스트.

## 결과

020의 HTTP 계약(상태/모양/errorCode)이 web slice로 고정됨.

## 요약

- `@WebMvcTest`(서비스 mock)로 201/400/거절 매트릭스/GET/list 검증.
- 각 거절을 status + problem+json + errorCode 세 축으로 단언.
